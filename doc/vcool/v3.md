### 2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？
- **问题1：阶跃星辰API的CER校验不通过问题**
  - 现象：用户录制的音频经常出现`CER_NOT_PASS`错误，提示字符错误率过高，即使朗读正确也偶尔出现校验失败
  - 破局：优化错误提示，明确告知用户需要准确朗读训练文本，同时在录制界面增加高亮提醒，降低用户操作失误率；后续切换到硅基流动平台，彻底避开了CER校验机制的限制。
- **问题2：第三方API服务不稳定问题**
  - 现象：阶跃API频繁出现503 Service Unavailable错误，服务可用性差，影响用户体验
  - 破局：快速评估备选方案，切换到硅基流动MOSS-TTSD-v0.5模型，重构接口对接逻辑，仅用1小时就完成了平台迁移，保证功能正常可用。
- **问题3：不同TTS平台的参数差异适配问题**
  - 现象：阶跃和硅基流动的API参数结构差异很大，需要同时兼容两种请求格式
  - 破局：抽象出统一的TTS服务接口层，内部适配不同平台的参数转换，上层业务代码不需要感知底层平台差异，为后续接入更多TTS平台打下基础。

## 四、本周随手记（选填）
- 第三方API对接一定要提前调研服务稳定性，最好准备备选方案，避免依赖单平台出现服务不可用的问题。
- API Key必须放在环境变量中，绝对不能硬编码在前端代码中，防止密钥泄露。
- 设计错误提示时要站在用户角度，不要直接抛出技术术语，比如把`CER_NOT_PASS`转换成"训练文本与录音内容不匹配，请准确朗读"，用户能立刻明白问题所在。
- 功能切换时尽量保持用户界面不变，只替换底层实现，用户学习成本为零，体验更流畅。
- 语音类功能要特别注意音频格式兼容性，WAV格式的通用性最好，优先使用。
- 录音功能在移动端需要特殊处理权限请求，要明确告知用户录音用途，提升用户信任度。
